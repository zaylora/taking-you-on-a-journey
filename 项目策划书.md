# AI 智能旅游规划 App — 项目策划书

> 一句话定位：一款**对话式**的 AI 旅游规划应用，用户用自然语言描述需求，系统通过多 Agent 协作自动生成可落地的逐日行程攻略，并在**高德地图上实时打点、查询与联动调整**。

- **文档版本**：v1.0
- **日期**：2026-06-17
- **技术栈**：后端 LangGraph + uv；前端 Vite + Vue 3 + Element Plus；地图 高德地图 API；LLM 海外模型（OpenAI / Claude）
- **阶段定位**：先交付 MVP，但分层与接口设计为未来产品化（用户系统、持久化、并发、运维）预留扩展点

---

## 一、项目背景与目标

### 1.1 痛点

传统旅游攻略生成存在三大问题：

1. **信息割裂**：景点、餐饮、交通、住宿、天气分散在不同平台，需用户手动整合。
2. **缺乏个性化排程**：现有工具多是"清单堆砌"，不考虑坐标聚类、顺路、天气、预算约束。
3. **不可交互**：生成后难以局部微调，用户想"把第二天的景点换一个"往往要重来。

### 1.2 产品目标

- 用户用对话表达需求（城市、日期、人数、偏好、预算）即可获得**结构化、逐日、可落地**的攻略。
- 攻略与地图**双向联动**：攻略项可在地图上定位；地图上选点可反向影响行程。
- 关键约束（预算、天气、顺路）由系统自动处理，超支/雨天等情况自动调整。

### 1.3 MVP 范围界定（Scope）

| 模块          | MVP 包含                    | 产品化阶段（预留）              |
| ------------- | --------------------------- | ------------------------------- |
| 对话式规划    | ✅ 多会话、真正多轮、流式输出 | 会话分享、团队协作              |
| 多 Agent 编排 | ✅ 完整 7 步编排            | Agent 可观测/可回放、A/B 提示词 |
| 地图联动      | ✅ 打点、POI 查询、点选定位、逐日顺路路线规划绘制 | 导航跳转          |
| 用户系统      | ⛔（匿名 session）          | 注册登录、攻略收藏与历史        |
| 持久化        | SQLite checkpointer + 会话元数据 | PostgreSQL + Redis              |
| 部署          | 本地单机                    | 容器化、CI/CD、监控告警         |

### 2.2 核心交互流程（对话为主 + 地图实时联动）

1. **需求收集**：用户在对话框输入"想去成都玩 3 天，2 个人，爱吃辣，预算人均 2000，喜欢历史古迹"。输入框上方提供**快捷要素 chips**（城市 / 日期 / 人数 / 预算），降低纯自然语言的歧义。
2. **需求澄清（多轮问答 / Plan 模式）**：系统先评估输入是否完整，对缺失或模糊处主动提问，并尽量**给出选项**让用户点选（如"出行节奏？[紧凑/适中/悠闲]"、"住宿档位？[经济/舒适/高端]"、"是否需要含往返大交通？"）。用户可点选项或自由作答；信息齐备后才进入下一步。此阶段不调用检索，避免"垃圾进垃圾出"。
3. **进度可视化**：澄清完成后，对话区展示 **Agent 编排进度**（任务派发 → 并行检索 → 行程编排 → 住宿 → 预算核算 → 汇总），每个节点流式回传中间状态。
4. **结果呈现**：右侧地图自动打点（按天用不同颜色），行程卡片按天展开；对话区给出文字版攻略与说明。
5. **多轮追问与联动调整**：
   - 用户点击"新建会话"创建一个新的对话框/行程线程，多个会话互相隔离，可切换恢复。
   - 对话追问"第二天太赶了，去掉一个景点" → 系统读取历史消息与上轮 `day_plans`，识别为修改请求并增量更新。
   - 对话追问"刚才那个行程适合带老人吗？" → 系统基于当前方案回答，不全量重跑。
   - 点击行程卡片中的某景点 → 地图平移高亮 + 弹出 POI 详情。
   - 在地图上点选一个新 POI → "替换/新增到第 X 天"，触发**局部重排**（仅重排受影响的天，不全量重跑）。

### 3.1 架构关键决策

| 决策点         | 选型                         | 理由                                                            |

## 四、后端 Agent 编排详细设计（LangGraph）

> 严格对应你的《旅游攻略 Agent 编排-改进版》流程图。

### 4.1 全局状态 State 设计

```python
from typing import TypedDict, Annotated
from operator import add

class TripState(TypedDict):
    # —— 输入 ——
    raw_input: str                 # 用户原始自然语言
    city: str
    start_date: str                # ISO 日期
    end_date: str
    num_people: int
    preferences: dict              # {travel, accommodation, food}
    budget: float                  # 总预算或人均

    # —— 需求澄清（多轮问答）——
    clarify_history: list          # 澄清问答历史 [{question, options, answer}]
    missing_fields: list           # 仍缺失/模糊的字段
    clarified: bool                # 是否已澄清完毕

    # —— 任务派发产出 ——
    normalized_req: dict           # 标准化改写后的结构化需求

    # —— 并行检索产出（各 Agent 写入各自字段，避免写冲突）——
    weather: dict                  # 预报或降级气候
    attractions: list              # 候选景点 + 坐标
    restaurants: list              # 候选餐厅池
    transport: dict                # 大交通 + 市内交通

    # —— 行程编排产出 ——
    day_plans: list                # 逐日聚类排程结果
    daily_centers: list            # 每天活动中心点（供住宿用）

    # —— 住宿产出 ——
    hotels: list

    # —— 预算核算 ——
    budget_check: dict             # {total, breakdown, over_budget: bool}
    retry_count: int               # 回退重排计数（防死循环）

    # —— 汇总 ——
    final_plan: dict               # 最终结构化攻略

    # —— 过程消息（流式进度）——
    messages: Annotated[list, add]
```

### 4.2 节点定义与职责

| 节点 ID         | 名称              | 职责                                                                   | 调用工具/LLM               |
| --------------- | ----------------- | ---------------------------------------------------------------------- | -------------------------- |
| `clarify`       | ①.5 需求澄清 ★    | Plan 模式与用户多轮问答；对缺失/模糊处提问（可给选项）；澄清完成才放行 | LLM（强模型）+ `interrupt` |
| `dispatch`      | ② 任务派发 Agent  | 需求标准化改写，拆解为子任务                                           | LLM（强模型）              |
| `weather`       | ③d 天气查询       | 取预报；远期降级为季节/历史气候                                        | 高德天气 API               |
| `attractions`   | ③a 游玩地检索     | 候选景点+坐标，不够则补搜（带退出条件）                                | 高德 POI + LLM             |
| `restaurants`   | ③b 餐饮检索       | 按饮食偏好建候选餐厅池                                                 | 高德 POI（+点评）          |
| `transport`     | ③c 交通检索       | 大交通 + 市内交通方案                                                  | 高德路径/搜索              |
| `itinerary`     | ④ 行程编排 ★      | 坐标聚类→分天→顺路排序，天气调整，餐厅就近分配                         | LLM + 算法                 |
| `accommodation` | ⑤ 住宿 Agent      | 按每天活动中心点选酒店                                                 | 高德 POI/周边              |
| `budget_check`  | ⑥ 预算核算+校验 ★ | 门票+住宿+餐饮+交通是否超支，schema 校验，缺失补偿                     | 算法 + LLM                 |
| `summarize`     | ⑦ 汇总 Agent      | 按已排结构写每天攻略，输出最终攻略                                     | LLM（强模型）              |

### 4.3 图结构（含并行与回退）

```python
from langgraph.graph import StateGraph, START, END

g = StateGraph(TripState)

g.add_node("clarify", clarify_agent)
g.add_node("dispatch", dispatch_agent)
g.add_node("weather", weather_agent)
g.add_node("attractions", attractions_agent)
g.add_node("restaurants", restaurants_agent)
g.add_node("transport", transport_agent)
g.add_node("itinerary", itinerary_agent)
g.add_node("accommodation", accommodation_agent)
g.add_node("budget_check", budget_check_agent)
g.add_node("summarize", summarize_agent)

# 入口先进入“需求澄清”阶段
g.add_edge(START, "clarify")

# 条件边：信息不足 → 自身循环继续追问；澄清完成 → 进入任务派发
def route_after_clarify(state: TripState) -> str:
    return "dispatch" if state["clarified"] else "clarify"

g.add_conditional_edges("clarify", route_after_clarify,
                        {"clarify": "clarify", "dispatch": "dispatch"})

# 并行派发：dispatch 完成后同时触发 4 个检索节点
g.add_edge("dispatch", "weather")
g.add_edge("dispatch", "attractions")
g.add_edge("dispatch", "restaurants")
g.add_edge("dispatch", "transport")

# 4 个并行节点全部完成后汇入 itinerary（LangGraph 自动等待入边）
g.add_edge("weather", "itinerary")
g.add_edge("attractions", "itinerary")
g.add_edge("restaurants", "itinerary")
g.add_edge("transport", "itinerary")

g.add_edge("itinerary", "accommodation")
g.add_edge("accommodation", "budget_check")

# 条件边：超支 → 回退重排（带重试上限）；否则 → 汇总
def route_after_budget(state: TripState) -> str:
    if state["budget_check"]["over_budget"] and state["retry_count"] < 2:
        return "itinerary"      # 回退重排
    return "summarize"

g.add_conditional_edges("budget_check", route_after_budget,
                        {"itinerary": "itinerary", "summarize": "summarize"})

g.add_edge("summarize", END)

app = g.compile()
```

**关键设计点**：

- **需求澄清（Plan 模式）**：入口节点 `clarify` 先评估输入完整度，发现缺失/模糊字段时通过 LangGraph `interrupt` 暂停图执行，向前端抛出问题（可附带候选选项）；用户回答后用 `Command(resume=...)` 恢复执行。信息齐备前在 `clarify` 自循环，齐备后才进入 `dispatch`，避免"垃圾进垃圾出"。详见 4.6。
- **并行**：4 个检索节点共享同一上游 `dispatch`，LangGraph 自动并行执行；各自写入 State 的独立字段，**避免写冲突**。
- **天气影响排程**：`weather` 结果传入 `itinerary`，雨天将室外景点替换为候选室内项。
- **超支回退**：`budget_check` 通过条件边回到 `itinerary`，`retry_count` 限制最多回退 2 次，防死循环；超过上限则带"已尽力压缩"的说明进入汇总。
- **退出条件**：`attractions` 补搜需设最大轮次（如 ≤3 次）与最少候选数阈值；`clarify` 也需设最大问答轮次，超过则用默认值兜底继续。

### 4.4 需求澄清机制（Human-in-the-Loop / Plan 模式）

入口节点 `clarify` 是与用户多轮交互的"计划"阶段，借助 LangGraph 的 `interrupt` + checkpointer 实现"提问→暂停→等用户回答→恢复"。

```python
from langgraph.types import interrupt, Command

def clarify_agent(state: TripState) -> dict:
    # 1) LLM 评估当前需求完整度，找出缺失/模糊字段
    gaps = assess_requirements(state)          # -> [{field, question, options?}]
    if not gaps:
        return {"clarified": True}

    # 2) 取第一个缺口，向前端抛出问题（可带候选选项），暂停执行
    gap = gaps[0]
    answer = interrupt({
        "type": "clarify",
        "field": gap["field"],
        "question": gap["question"],
        "options": gap.get("options", []),     # 给用户选择项（可为空=自由文本）
    })

    # 3) 用户回答后恢复，写回 state，继续下一轮（由条件边再次进入 clarify）
    return {
        "clarify_history": [{**gap, "answer": answer}],
        "clarified": False,
    }
```

**前端恢复方式**：图在 `interrupt` 处暂停并通过 SSE 把问题（含选项）推给前端；用户在对话区作答或点选项后，前端调 `/api/chat`（携带 `thread_id`）触发 `app.invoke(Command(resume=用户回答), config)` 继续。

**前端展示**：澄清问题渲染为对话气泡 + Element Plus 选项按钮（`el-radio-group`/`el-button`），用户点选即作答；也允许直接打字自由回答。

**要点**：

- 必须配置 **SQLite checkpointer** 并为每次会话分配 `thread_id`，`interrupt` 和普通多轮追问才能跨请求、跨后端重启恢复。
- 设最大澄清轮次（如 ≤4 轮），超过则对剩余字段取合理默认值并继续，避免无限追问。
- 选项由 LLM 依据上下文生成（如预算档位、出行节奏快/慢、亲子/情侣/独行），降低用户输入成本。

### 4.5 真正多轮对话机制

MVP 不只保存 `thread_id`，还要求历史真正参与后续推理。每个会话的 LangGraph state 通过 SQLite checkpointer 持久化，核心上下文包括：

- `messages`：用户与 AI 的消息流，用于理解“刚才”“第二天”“那个酒店”等引用。**攻略全文不进 `messages`，只存一句简短回执**（如“已生成成都 3 天行程，详见行程面板”）；完整行程由 `day_plans` 承载，避免一条消息塞进整份攻略撑爆上下文。
- `conversation_summary`：长会话压缩摘要，只保留偏好、已确认约束、否定过的方案与未决事项；**行程本身不进摘要**（由 `day_plans` 全量保存）。近期原文按 token 预算保留，超出部分才压缩成摘要。
- `day_plans` / `budget_check` / `normalized_req`：上一版结构化行程、预算与需求，由 checkpointer 全量保存，供修改和问答复用。
- `last_intent` / `refine_request`：本轮意图与结构化修改指令。

后端在常规编排前增加意图识别：

| 意图 | 触发场景 | 行为 |
| ---- | -------- | ---- |
| `plan_new` | 首次规划、明确换城市/重新规划 | 进入 `clarify → dispatch → 检索 → 编排` 全链路；在已有会话内重规划时先重置上一版的澄清/预算/行程状态，避免脏数据沿用 |
| `refine_existing` | “第二天少一点”“预算改成 3000”“酒店换近一点” | 由 `refine` 节点**直接在旧 `day_plans` 上局部编辑**，只动受影响的天/字段、不回到全量编排；仅当需要新 POI 时才补检索 |
| `qa` | “适合带老人吗”“为什么这么排” | 基于历史与当前方案回答，不改行程、不重跑检索 |

这样用户的第二轮、第三轮不会变成孤立请求，而是基于当前会话状态继续工作。

### 4.6 流式进度上报

利用 `app.astream_events(...)` 监听节点开始/结束事件，映射为前端进度：

```python
async for event in app.astream_events(initial_state, version="v2"):
    kind = event["event"]
    if kind == "on_chain_start":
        yield sse({"type": "node_start", "node": event["name"]})
    elif kind == "on_chain_end":
        yield sse({"type": "node_end", "node": event["name"]})
    elif kind == "on_chat_model_stream":   # 汇总阶段逐字输出
        yield sse({"type": "token", "content": event["data"]["chunk"].content})
```

### 4.7 提示词策略（海外模型）

- **结构化输出**：所有中间 Agent 用 **function calling / structured output**（Pydantic schema）强约束返回 JSON，避免解析失败。
- **中文输出**：汇总 Agent 的 system prompt 显式要求**简体中文**输出。
- **成本控制**：检索类节点用较便宜模型（如 GPT-4o-mini / Claude Haiku），编排与汇总用强模型（GPT-4o / Claude Sonnet）。此为产品化阶段的"混合模型"优化预留接口。

### 6.2 API 接口

## 八、开发计划（里程碑）

> 建议按"链路先打通，再做体验"的顺序，避免前后端长期分离。

### M1：骨架与链路打通（基础设施）

- 后端 FastAPI + uv 工程；前端 Vite + Element Plus 骨架。
- 跑通一个**最简单的 LangGraph**（dispatch → summarize 直连）+ SSE 流式到前端对话框。
- **验收**：用户输入一句话，前端能流式看到 AI 回复。

### M2：核心 Agent 编排

- 实现 `clarify` 需求澄清节点（`interrupt` + checkpointer + `thread_id`），跑通"提问带选项→暂停→恢复"的多轮问答。
- 实现 4 个并行检索 Agent（先用高德 POI/天气真实 API）。
- 实现 `itinerary` 行程编排（坐标聚类 + 分天 + 顺路）。
- **验收**：输入模糊需求时能多轮追问澄清；澄清完成后产出结构化 `day_plans`（暂不接地图 UI）。

### M3：地图与行程联动

- 高德地图打点、聚合；行程卡片渲染。
- 卡片 ↔ 地图双向联动；POI 后端代理。
- **验收**：生成行程后地图自动打点，点击卡片地图高亮。

### M4：住宿 + 预算闭环

- `accommodation` + `budget_check`（含超支回退条件边）。
- 前端展示预算明细与超支提示。
- **验收**：完整 7 步编排跑通，超支能自动重排。

### M5：真正多轮对话 + 多会话持久化

- 前端新增会话列表与"新建会话"按钮，点击后生成新的对话框/`thread_id`；`activeThreadId` 落 localStorage，刷新后恢复上次会话。
- 后端新增 `/api/sessions`，会话元数据落 SQLite（`session_meta` 表，含软删除）；会话存在性以该表为准。
- LangGraph checkpointer 从 `MemorySaver` 升级为 SQLite（`AsyncSqliteSaver` 在 FastAPI lifespan 内创建并随 graph 挂到 `app.state`），支持刷新和后端重启后恢复。
- 新增意图识别与多轮上下文：`plan_new` / `refine_existing` / `qa`（澄清恢复在 API 层拦截，先于意图识别）。
- 支持基于上一版 `day_plans` 的自然语言修改（如"第二天轻松一点"、"预算改成 3000"、"换一家近地铁的酒店"），由 `refine` 节点局部编辑、不全量重排。
- **验收**：多个会话互不串；后端重启后能继续旧会话；用户能追问、问答、修改上一版行程，且修改只动受影响的天。

### M6：路线规划（行程地理质量 + 路线绘制）

> 由原"产品化预留：路线规划绘制"提为 MVP。根因：行程点本身顺序烂、餐厅离景点几十公里、交通段由 LLM 瞎编，前端只是忠实画出烂数据。详见 `docs/superpowers/specs/2026-06-20-route-planning-m6-design.md`。

- 后端重写 `itinerary` 结构生成：**算法主导**「分天 + 顺路 + 就近餐厅 + 交通段」，LLM 降级为只填软字段（花费/备注/室内/时间），不能改顺序/坐标。
- 新增 `amap.search_around`（高德周边检索）：按每天簇中心就近搜餐厅，消灭"折返市中心吃饭"的大回环。
- 交通段由算法在每对相邻点间插入：真实起讫坐标（沿用相邻点）+ 真实 from/to + 按直线距离定方式（<1km 步行 / 1~5km 公交 / >5km 驾车）。
- 前端最小适配：数据修好后分段总览路线自动连续、按天配色、无 7km 步行怪线。
- **验收**：餐厅贴当天景点、每跳有段且方式合理、总览路线连续不折返。

### M7：交互打磨与导出

- Agent 进度指示器细节打磨、快捷要素 chips。
- 地图点选新增/替换 → 局部重排体验完善。
- 攻略导出。
- **验收**：可演示的完整 MVP。

### M8（产品化预留，非 MVP）

- 用户系统、PostgreSQL/Redis 持久化、容器化部署、限流监控。

## 九、关键风险与应对

| 风险                      | 影响         | 应对                                               |
| ------------------------- | ------------ | -------------------------------------------------- |
| LLM 返回非结构化/解析失败 | 编排中断     | 强制 structured output + Pydantic 校验 + 重试      |
| 高德 API 额度/限流        | 检索失败     | 后端缓存 + 限流 + 失败降级（如天气降级为季节气候） |
| 并行节点写 State 冲突     | 数据错乱     | 各节点写**独立字段**，不写同一 key                 |
| 超支回退死循环            | 请求挂起     | `retry_count` 上限 + 兜底说明                      |
| 海外 LLM 网络不稳定       | 响应慢/超时  | 后端设超时、重试、（产品化）接入网关               |
| 地图 Key 泄露             | 额度被盗刷   | Web 服务 Key 仅后端持有；JS Key 配域名白名单       |
| 长链路耗时长              | 用户等待焦虑 | 流式进度可视化 + 各节点 partial 结果先行展示       |
